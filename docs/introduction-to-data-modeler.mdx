The Data Modeler intends to support self-service data model changes in your Kubit environment. Want to add handling for a new column in your fact table? Create a derived property, add a new dimension table, or even a whole new schema? This article will provide an introduction to the Data Modeler and will help you understand the main concepts behind it and what you can do in it.

## Prerequisites

* Full access to the Data Modeler feature is exclusively available to [Direct Connect](doc:data-warehouse) customers. Customers utilizing [Data Share](doc:secure-data-sharing) and Managed Service have read-only access to the Connection and Tables tabs.
* You have to be a Kubit Admin to access Data Modeler.

# Getting Started

You can find the Data Modeler icon in the "Data Management" icon on the bottom left of your screen.

![](/images/docs/b9b53e2dbaa3cfac5ea770b34a0d4fc1e83887828a1ea9ab6479203b9b90a16f-data_management_nav.gif)

Once you click on it you will see:

* A `New Schema` button you can use to create a new schema.

* A list of existing Kubit Schemas on the left.

* The default Schema configuration which you can observe or edit by navigating through the tabs:

  * `Configuration` - The primary configuration for this Schema and area where you can update Schema names and add Schema descriptions.
  * `Connection` - The connection parameters for the schema to your cloud data warehouse.
  * `Tables` - The list of all tables/views being used in your Kubit schema. This is also the area where you can add more tables, expose more fields, and add derived columns.
  * `Fields` - The "semantic" layer for Kubit and area you can rename Fields, add Field descriptions, create derived fields, and control if a field is available for breakdown and filtering. You can also disable Fields here so they no longer appear in Kubit.
  * `Optimization` - An area where you can set query timeout parameters, look back windows, and Sampling rates for on-demand sampling in Kubit.
  * `View Model` - View your entire data model.
  * `Version History` - Review past versions of the Schema and restore old versions if necessary.

  ![](/images/docs/8a04435b31fdea3fb4c7cfc1ea7c6e11be07a267b7f634fb7328e6bd6c6ab2cd-Screenshot_2025-10-24_at_8.37.47.png)

# Configuration

In this view, you can see:

* `Schema Description`
  * Users with permissions in both Direct Connect and Data Share setups can update the Schema Description
* What `Timezone` is being used for this Schema.
* When doing reports with a `Time Unit: Weekly` reports - what day of the week is the Week Start set to.

This is where you will see all of these parameters in action when you are building reports:

## Subjects

In Kubit, the Subject of your data is what is analyzed. A read-only list of all [Subjects](doc:components-of-a-report#subject) can be configured for the schema.

## Sampling

Sampling can be used to speed up reports and to save costs, especially when dealing with very large datasets. To understand more about Sampling in Kubit you can read this [detailed article](doc:sampling-options#how-to-sample-in-kubit). What you can control from here is the default sampling rates for:

* **Default**: applies to all Query/Funnel/Flow/Retention reports.
* **Cohort**: applies to all Cohort Filters and Segments.

![](/images/docs/6eb4ea9a990646315d61c4c5a6a9c0ccd2ae82bf4fe681e68d6600465796f9f1-Sampling_cohort.png)

## Minimum Refresh Interval

The Minimum Refresh Interval (MRI) dictates how often Events and Fields can be refreshed. Users with a Creator, Governor, or Admin role can refresh these Events or Fields. For example, if the MRI is set to one day, only one user can refresh it per day. Other users who try to refresh it within that time frame will not be able to until the MRI is up.

All Events can be refreshed from the Event drop down.

![](/images/docs/ecc6ea848fbb8e8704369459dbae6656c4ceeeb85358744cabae88defd2f8418-minimal_refresh_interval.png)

Field values can be refreshed by selecting a specific measure filter or report filter, and then refreshing the values at the bottom.

![](/images/docs/174b082e33178d1fdb0e85c154e942bd965eb18975bfce15ed86c7762741a0af-minimum_refresh_iterval_2.png)

Since the values are only updated when a user initiates a refresh, this reduces the computational burden on your warehouse.

## Additional Funnel and Flow Partition Fields

The default `Partition` options in Kubit are:

* None.
* Day.
* Conversion Window.

More about Partitions is located [here](doc:funnel#partition-conversion-window).

In a nutshell, the `Partition` options control the time window within a conversion that happened within a Funnel or a Flow. But in some cases, you might be interested in using a non-temporal field for the partitioning such as Session. This then allows you to configure the Funnel report to measure only the conversions that happened within a single session.

![](/images/docs/3a3208032c9b79b59a72c4dfb887aa6d4bd51bcbde36961fb3ac8957a943e31d-Additional_funnel_and_path_partition_fields.png)

## Status

The Enabled toggle controls whether a Schema is visible for the Kubit users who build Reports or not. Once disabled, users can no longer select this schema from any Schema dropdown.

<Info>
  ### Disable a schema while still working on it

  Set the status to `Disabled` when you're working on a new Schema and don't want your teammates to start using it just yet.
</Info>

# Connection

In order to consume data from your data warehouse you need to set up the connection to it. Kubit relies on a JDBC connection to execute [Reports](doc:introduction-to-the-main-navigation-items#reports).

<Warning>
  ### Limited Access

  This functionality is only available to Direct Connect customers and for users with role `Admin`.
</Warning>

The first step is to choose the warehouse type from a list of supported warehouses:

* Snowflake
* BigQuery
* Databricks
* ClickHouse
* MySQL
* Presto

Then, you have to fill out the rest of the parameters which are as follows (with the exception of BigQuery):

* **Username**: which should be used to login to your warehouse (we recommend creating a dedicated user for Kubit).
* **Password**: for the username (we recommend a length of 60 characters, a randomly generated mix of uppercase, lowercase, numbers, and special characters).
* **Timeout**: on the JDBC connection, make sure to check what the timeout configured on your warehouse and set it to the same value or less.
* **JDBC URL**: to establish the JDBC connection.
* **End Date Type**: what should the date picker default to as the end date - today, yesterday, a static date, or something else? The recommendation is to keep the default (`CURRENT`).

## Create a Derived Column

Derived Columns come to the rescue when you don't have all of the required columns already in place or you want to slightly modify the values in a column. A Derived Column, in a nutshell, allows you to create a new logical column based on other columns that already exist in the table. To start, click on `Add Derived Column`:

Then you have 2 ways to define the column:

1. **Raw SQL**: most common use cases are when you need to cast a type, concatenate strings, or pick a property from an object, e.g., JSON or Struct.
2. **Mapped String**: useful when you need to map an enum, sometimes the database would utilize a numeric type to store some data and you need the human-readable names in your analytics tool - this is when you will use Mapped String.

<Warning>
  ### Only row operations in SQL are supported

  The SQL option supports only single-row operations. If you'd like to use Window Functions, please check out Derived Fields.
</Warning>

Here are a couple of examples, one is using SQL to create a date column from a timestamp column:

And the other is mapping integers to human-readable names:

![](/images/docs/eca1c17e2a53b8183214e90559dbf3fa4eac6d20e06d28c5022dd80cf2f0514a-create_a_derived_column_2.png)

## Derived Fields

There are 2 main scenarios in which you should use `Derived Field` instead of a `Derived Column`:

1. You want to apply SQL Window Functions, e.g. to find out the duration of a session by calculating the time difference between the first and last event in a session.
2. You have multiple fact tables which contain the same`Field` so instead of defining `Derived Column` in each table, you can define it as a `Derived Field` only once.

To go back to our session duration example, let's check the `Window Functions` checkbox and input this sample SQL:

```
timestampdiff(second, min(@event_timestamp) over (partition by @session_id), max(@event_timestamp) over (partition by @session_id))
```

<Warning>
  ### Referencing a column

  Remember to use @ when referencing a column!
</Warning>

![](/images/docs/80c20eb961dc552cb3ca8be937dafa94469c0546a50e14eb48eba2018b92d158-derived_fields.png)

When `Window Functions` aren't being used the behavior is equivalent to `Derived Column`, it just may be applied over more than 1 column in case you have multiple fact tables in a union.

# Version History

Direct Connect customers are able to see all previous versions of your Kubit Data Model, what changes were made, and who made them. Additionally when a new version of a schema is published you will be prompted to provide a description of the changes for better lineage.

## See Previous Data Model Versions

* Navigate to the Data Modeler section of Kubit and select the Schema you want to review.
* Click the new "Version History" in the top navigation of the Data Modeler
* You will now see a list of all previous versions and changes made.

![](/images/docs/5a9f08865dbacc092291c187006aab21d90fb58a731d7260d960d2c06cb44577-see_previous_data_model_versions.gif)

## Restore Previous Data Model Version

If you make a change to a data model and need to restore to a previous version you're able to do this in the "Version History" section of the data modeler.

* Identify the version you want to restore to
* Click the 🔄 icon next to "Archived" on the row of the data model you want to restore.
* Type "confirm" in the text box and click the Confirm button
* The previous version will be enabled and replace the existing data model

![](/images/docs/819d0ac15f5aaf09fa941c90550700ffccfd20e471480713e266a569a541b082-restore_previous_data_model_version.gif)

## Things to be mindful of when restoring a previous data model version

Existing reporting using any Subjects or data from a table that is removed when a data model version is restored will still load but cannot be re-executed.

Restoring a data model version that eliminates Subjects could create instances of reports showing an "Unknown" subject. This is how Kubit will represent disabled subjects for already executed reports.

Changes are instantaneous so if a user is executing a report while the version is being restored and that report contains data that is being removed could cause an error.
